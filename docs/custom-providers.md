# 自定义 LLM Provider 维护指南 | Maintaining Custom LLM Providers

> 适用范围 | Applies to: `develop` 及之后的版本（自 `feat/hardcode-providers` 起）
>
> 最后更新 | Last updated: 2026-04-18

---

## 中文

### 为什么要硬编码？

此前后端提供了 `POST /v1/providers/custom` 端点，允许前端 UI 动态添加 OpenAI-compatible provider。该实现存在两个根本问题：

1. **Dev 模式下写文件逻辑脆弱**：通过字符串拼接直接修改 `core/kortix-master/opencode/opencode.jsonc`，多次增删后会破坏 JSONC 缩进和逗号，最终导致沙盒启动时报 `ConfigJsonError`。
2. **Prod（installed）模式下根本走不通**：代码分支依赖 `findRepoRoot()`，prod 环境下没有仓库根，直接返回 `501 Not Implemented`，即"装上去用户看不到这个功能"。

综合考虑"终端用户不应自定义 provider"这一产品诉求，我们选择**一律硬编码**：

- Provider 清单写在 `core/kortix-master/opencode/opencode.jsonc` 的 `"provider"` 块里
- 对应 API Key 通过 `apps/api/.env` 的环境变量注入
- 前端"添加自定义 Provider"入口已移除；后端 `POST/DELETE /v1/providers/custom` 统一返回 `410 Gone`

### Env 文件分发链路（重要）

**根 `.env` 是唯一真相源**。`scripts/setup-env.sh` 读取根 `.env` 后自动生成三个下游文件：

```
.env (ROOT — 源，人工编辑这里)
   │
   ▼  scripts/setup-env.sh
   ├──► apps/api/.env        ← API 进程读
   ├──► apps/web/.env        ← 前端读 NEXT_PUBLIC_*
   └──► core/docker/.env     ← 沙盒容器（由 docker-compose.yml 通过 env_file 加载）
```

**下游 3 个文件不要手动编辑**——开头都有 `Auto-generated by scripts/setup-env.sh — do not edit manually`，下次跑脚本会被覆盖。

OpenCode 在沙盒里读 `{env:XXX_API_KEY}` 时，变量来自 **`core/docker/.env`**（通过容器环境），所以新增 provider 时必须同步 `setup-env.sh` 里那个块。

### 如何新增 / 修改 / 删除一个 Provider

1. **编辑 `core/kortix-master/opencode/opencode.jsonc`** 的 `"provider"` 块，按现有条目的格式添加：

   ```jsonc
   "myprovider": {
     "name": "My Provider",
     "npm": "@ai-sdk/openai-compatible",
     "options": {
       "baseURL": "https://api.example.com/v1",
       "apiKey": "{env:MYPROVIDER_API_KEY}",
     },
     "models": {
       "my-model": {
         "name": "My Model",
         "id": "my-model-id",
       },
     },
   },
   ```

2. **在 `scripts/setup-env.sh`** 的 `write_env "core/docker/.env"` 块里加一行（**这步最容易忘**）：

   ```bash
   "$(kv MYPROVIDER_API_KEY)" \
   ```

3. **在 `apps/api/.env.example`** 对应区块添加占位，方便同事知道要填什么：

   ```
   MYPROVIDER_API_KEY=                     # My Provider 描述
   ```

4. **在根 `.env`** 填入真实 key：

   ```
   MYPROVIDER_API_KEY=sk-xxxx
   ```

5. **重新分发 env + 重启沙盒**：

   ```bash
   bash scripts/setup-env.sh        # 把 key 写进 core/docker/.env
   pnpm dev:sandbox                 # 让沙盒带上新 env 重启
   ```

   `opencode.jsonc` 通过 bind mount 在沙盒内实时可见，但 OpenCode runtime 只在启动时解析一次，所以改完 provider 必须重启沙盒（或调 `POST /v1/core/restart/opencode-serve`）。

### 排错速查

| 现象 | 可能原因 | 查什么 |
| --- | --- | --- |
| 沙盒 `ConfigJsonError` | `opencode.jsonc` 有语法错 / 手搓拼接残留 | `cat core/kortix-master/opencode/opencode.jsonc` 用 VSCode JSONC 插件看 |
| 请求发不出、OpenCode 报空 key | `core/docker/.env` 里没这个 key | `grep MYPROVIDER core/docker/.env` 是否存在且非空 |
| 根 `.env` 有 key 但沙盒还是空 | 没跑 `setup-env.sh` / 没重启沙盒 | 重跑上面第 5 步 |
| `setup-env.sh` 跑完沙盒 env 仍是旧值 | `setup-env.sh` 里漏加 `$(kv XXX)` | 检查 `scripts/setup-env.sh` 的 `write_env "core/docker/.env"` 块 |

### 约定

- **不要** 手工修改生产环境沙盒里的 `opencode.jsonc`——配置应来自 repo，便于团队共享与版本控制。
- **不要** 重新启用前端 Custom Provider 入口或 `POST /custom` 端点——除非先用 `jsonc-parser` 库重写写入逻辑，或迁移到 OpenCode 官方 `client.global.config.update()` 路径（见 `suna-latest` upstream commit `0a31da48b`）。
- Provider ID 仅允许 `[a-zA-Z0-9_-]`，避免破坏 JSONC 或 env 变量名。
- **API Key 的环境变量命名** 保持和 `opencode.jsonc` 中 `{env:XXX_API_KEY}` 完全一致，否则沙盒启动时该 provider 会报空 key。

### 当前已硬编码的 Provider

| Provider ID | 说明 | 需要的 env key |
| --- | --- | --- |
| `kortix` | Kortix 默认托管 provider（含 MiniMax / GLM / Kimi） | `KORTIX_TOKEN`, `KORTIX_API_URL`（自动注入） |
| `bigmodel` | GLM 官方（open.bigmodel.cn） | `BIGMODEL_API_KEY` |
| `claude` | Claude 中转（apipool.dev） | `CLAUDE_API_KEY` |

`opencode.jsonc` 里还有 `openai` / `deepseek` / `moonshot` 三个注释掉的示例。要启用时，取消注释 + 填对应 env key。

## English

### Why hardcode?

The legacy `POST /v1/providers/custom` endpoint had two root problems:

1. **Fragile JSONC writer in dev**: it edited `core/kortix-master/opencode/opencode.jsonc` via naive string splicing. Repeated insert/delete would break indentation and trailing commas, eventually tripping `ConfigJsonError` on sandbox startup.
2. **Broken in prod**: the code path depends on `findRepoRoot()`, which returns `null` in an installed/packaged deployment — so the endpoint just returned `501 Not Implemented`. End users never had a working "Add provider" feature.

Combined with the product decision that **end users should not be able to add providers**, we hardcode the full registry:

- Provider list lives in `core/kortix-master/opencode/opencode.jsonc` under `"provider"`
- API keys flow in via `apps/api/.env`
- The "Custom Provider" button in the web UI is removed; `POST/DELETE /v1/providers/custom` now return `410 Gone`.

### Env file distribution chain (important)

**The root `.env` is the single source of truth.** `scripts/setup-env.sh` reads it and regenerates three downstream files:

```
.env (ROOT — source; edit this by hand)
   │
   ▼  scripts/setup-env.sh
   ├──► apps/api/.env        ← read by the API process
   ├──► apps/web/.env        ← read by the web frontend (NEXT_PUBLIC_*)
   └──► core/docker/.env     ← loaded by the sandbox via env_file in docker-compose.yml
```

**Never hand-edit the three downstream files** — each has `Auto-generated by scripts/setup-env.sh — do not edit manually` at the top, and the next run of the script overwrites your changes.

When OpenCode inside the sandbox resolves `{env:XXX_API_KEY}` from `opencode.jsonc`, the variable is read from the container's environment, which comes from `core/docker/.env`. So adding a provider means keeping `setup-env.sh` in sync too.

### Adding / changing / removing a provider

1. **Edit `core/kortix-master/opencode/opencode.jsonc`** `"provider"` block, following the existing entries:

   ```jsonc
   "myprovider": {
     "name": "My Provider",
     "npm": "@ai-sdk/openai-compatible",
     "options": {
       "baseURL": "https://api.example.com/v1",
       "apiKey": "{env:MYPROVIDER_API_KEY}",
     },
     "models": {
       "my-model": {
         "name": "My Model",
         "id": "my-model-id",
       },
     },
   },
   ```

2. **Add a line in `scripts/setup-env.sh`** inside the `write_env "core/docker/.env"` block (**this step is the easiest to forget**):

   ```bash
   "$(kv MYPROVIDER_API_KEY)" \
   ```

3. **Add a placeholder in `apps/api/.env.example`** so teammates know what to fill:

   ```
   MYPROVIDER_API_KEY=                     # description of My Provider
   ```

4. **Fill the real key in the root `.env`**:

   ```
   MYPROVIDER_API_KEY=sk-xxxx
   ```

5. **Redistribute env + restart the sandbox**:

   ```bash
   bash scripts/setup-env.sh        # writes the key into core/docker/.env
   pnpm dev:sandbox                 # restarts the sandbox so it picks up the new env
   ```

   `opencode.jsonc` is bind-mounted live, but OpenCode parses it only at startup — any provider change therefore requires a sandbox restart (or a `POST /v1/core/restart/opencode-serve` call).

### Troubleshooting quick reference

| Symptom | Likely cause | What to check |
| --- | --- | --- |
| Sandbox reports `ConfigJsonError` | Syntax error in `opencode.jsonc` or leftover string-splicing corruption | Open `core/kortix-master/opencode/opencode.jsonc` in a JSONC-aware editor |
| Requests fail, OpenCode sees an empty key | The key is missing from `core/docker/.env` | `grep MYPROVIDER core/docker/.env` — present and non-empty? |
| Root `.env` has the key but the sandbox still sees empty | `setup-env.sh` not re-run, or sandbox not restarted | Rerun step 5 above |
| `setup-env.sh` ran but the sandbox env is still stale | `setup-env.sh` is missing a `$(kv XXX)` line for that key | Inspect the `write_env "core/docker/.env"` block in `scripts/setup-env.sh` |

### Conventions

- **Never** hand-edit `opencode.jsonc` inside a running sandbox — configuration must come from the repo so the team shares it via Git.
- **Do not** re-enable the Custom Provider UI entry or re-open `POST /custom` until the writer is rewritten via `jsonc-parser` (round-trip-safe) or migrated to OpenCode's own `client.global.config.update()` (see upstream commit `0a31da48b` in `suna-latest`).
- Provider IDs must match `[a-zA-Z0-9_-]` to avoid breaking JSONC keys or env var names.
- The env var used in `{env:XXX_API_KEY}` must match the real variable name exactly; otherwise the provider will start with an empty key.

| --- | --- | --- |

| `bigmodel` | GLM official (open.bigmodel.cn) | `BIGMODEL_API_KEY` |
