清楚了。核心结论：沙盒是以 account 为单位分配的，不是以用户为单位。                                                                                                                                
                                                                                                                                                                                                       
  分配规则                                                                                                                                                                                             
                                                                                                                                                                                                       
  Alice 登录 → 选了一个 active account（比如"销售部"）                                                                                                                                                 
              │                                                                                                                                                                                        
              ▼                                                                                                                                                                                        
  ensure-sandbox 检查 kortix.sandboxes：                                                                                                                                                               
    1. 这个 account 有没有 status='active' 的沙盒？                                                                                                                                                    
       有 → 直接用                       ← 最常见                                                                                                                                                      
       没有 → 去第 2 步
                                                                                                                                                                                                       
    2. 这个 account 有没有 'stopped' / 'archived' 的沙盒？                                                                                                                                             
       有 → 启动它，改 status='active'   ← 二次启动                                                                                                                                                    
       没有 → 去第 3 步                                                                                                                                                                                
                                                            
    3. 从 warm pool 捞一个预热的（pool.grab）                                                                                                                                                          
       有预热的 → 分配给这个 account，改 account_id         
       没有 → 现场创建一个新的                                                                                                                                                                         
                                                                                                                                                                                                       
  kortix.sandboxes 表只有 account_id 列，没有 user_id——沙盒归账号所有，不归用户。                                                                                                                      
                                                                                                                                                                                                       
  具体到你的场景                                                                                                                                                                                       
                                                            
  假设：
  - 你（super_admin）：有 personal account A
  - 建了"销售部" account（共 5 人）                                                                                                                                                                    
  - 建了"研发部" account（共 3 人）
  - 员工 Alice 在销售部                                                                                                                                                                                
                                                            
  则机器上最多有这些沙盒（account 实际活跃才创建）：

  沙盒 1 → A (你 personal)              ← 你在 personal 工作时创建
  沙盒 2 → 销售部                        ← 销售部第一个人工作时创建
  沙盒 3 → 研发部                        ← 研发部第一个人工作时创建
  沙盒 4-? → warm pool (预热未分配)      ← 超管配置的预热数量                                                                                                                                          
  
  5 个销售员工 + 3 个研发员工共用 2 个沙盒（+ 大家可能有各自 personal 的沙盒）。不是 8 个员工 8 个沙盒。                                                                                               
                                                            
  多人共用一个沙盒的含义                                                                                                                                                                               
                                                            
  Alice 和 Bob 都在销售部 → 他们共享一个沙盒。这意味着：

  ┌──────────────────────┬──────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │         方面         │                                                     情况                                                     │
  ├──────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ 同时在里面跑 session │ ✅ 能，OpenCode 支持多 session 并发                                                                          │
  ├──────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ 看到对方的文件       │ ✅ 看得到（/workspace 是共享的）                                                                             │
  ├──────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ 看到对方的对话历史   │ ⚠️  取决于 session 可见性（可能按 session id 私有，但如果 session 列表不过滤，能看到别人的 session 并点进去） │
  ├──────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ CPU/内存竞争         │ ⚠️  同一个机器上跑，谁占满了 CPU 大家都卡                                                                     │
  └──────────────────────┴──────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

  这就是之前问你"要账号有何用"时的隐私缺口的根源——账号共享 = 沙盒共享 = 工作透明。                                                                                                                     
  
  Warm pool 是什么                                                                                                                                                                                     
                                                            
  超管能预创建一批空白沙盒放着（/admin/sandbox-pool 页面能配），新 account 第一次要沙盒时直接从 pool 捞，不用现场 docker build，启动从 1-2 分钟降到 5-10 秒。                                          
  
  配置：min=3，max=10                                                                                                                                                                                  
  → 空闲时 pool 里保持 3 个热机                             
  → 被消耗掉一个，后台 60 秒内自动补一个                                                                                                                                                               
  → 最多同时 10 个预热                                                                                                                                                                                 
                                                                                                                                                                                                       
  对你的部署意义：如果预计每天有几个新部门同时创建，把 min 调到 3-5；如果部门数量稳定、偶尔来新人，min 设 1 就够。                                                                                     
                                                            
  生命周期                                                                                                                                                                                             
                                                            
  - 不主动删：沙盒建了不会因为"闲置"被 Suna 自己删                                                                                                                                                     
  - 订阅过期会 archived（云端计费场景）                     
  - 用户主动停 / 删（在 /admin/sandboxes 里手动）                                                                                                                                                      
  - Pool 里的陈旧沙盒每 60 秒扫一次，过龄的销毁                                                                                                                                                        
                                                                                                                                                                                                       
  所以你长期会有一堆沙盒堆积，除非主动清理。生产环境建议搭个 cron 跑"30 天没用的 stopped 沙盒 → 删除"。
